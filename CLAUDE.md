# CLAUDE.md: Agentic AI Development Constitution for Multi-Tenant Accounting Platform

This document serves as the constitution for all AI agents working on the Multi-Tenant Web-Based Accounting Platform project. All agents MUST adhere to these guidelines.

## 1. Project Overview

**Goal:** To develop a scalable, secure, and customizable multi-tenant web-based accounting platform, starting with core accounting features and progressively integrating advanced functionalities inspired by TallyPrime.

**Tech Stack:**
*   **Frontend:** React.js, TypeScript, TailwindCSS
*   **Backend:** Node.js (TypeScript) / Python (FastAPI/Flask)
*   **Database:** PostgreSQL / MySQL (Shared Database, Separate Schemas)
*   **Infrastructure:** Cloud-native (Docker, Kubernetes)

## 2. Branch Policy

*   All development MUST occur on feature branches named `feature/<descriptive-name>`. For example, `feature/user-authentication`.
*   Feature branches MUST be merged into `dev` after successful code review and CI/CD checks.
*   Direct commits to `main` or `dev` are strictly forbidden.

## 3. Documentation Protocol

*   Every code change MUST be documented in `STATUS.md` in reverse-chronological order, detailing what was changed, why, and which files were affected.
*   `TASKS.md` MUST be updated to reflect progress on tasks. A task is considered complete ONLY when the implemented feature is verified as functioning correctly and passes all tests.
*   New features or significant changes MUST include updated architectural documentation (e.g., data models, API contracts) if applicable.

## 4. Tech Stack Conventions

*   **Frontend:** Adhere to React best practices, use functional components, and maintain a consistent component library. Ensure responsiveness and accessibility.
*   **Backend:** Follow RESTful API principles. Implement proper error handling, logging, and input validation. Use ORM/ODM for database interactions.
*   **Database:** All database changes (schema migrations) MUST be version-controlled. Data access MUST be abstracted through repositories or services.
*   **Security:** Implement secure coding practices. NEVER hardcode sensitive information. Ensure proper authentication and authorization checks for all API endpoints.

## 5. Hard Rules

*   **No Mock Data Ever:** All development and testing MUST use real backend data or realistic test data generated by dedicated test utilities. Mocking external services is allowed for unit testing, but not for integration or end-to-end testing of core features.
*   **Verified Functioning Before Checking a Box:** A task in `TASKS.md` is only marked as complete (`[x]`) after the implemented feature has been thoroughly tested and verified to be functioning as per requirements.
*   **Spec-Driven Development:** Agents are forbidden from inventing schema or API endpoints. All development MUST adhere to predefined specifications (e.g., data models, API contracts, product scope documents).
*   **Tenant Isolation:** Data for different tenants MUST NEVER be mixed or accidentally exposed. Strict tenant ID filtering MUST be applied to all data access operations.
*   **No Direct Database Access:** Backend services MUST interact with the database only through defined ORM/ODM or data access layers. Direct SQL queries should be minimized and reviewed.
*   **Performance:** All code MUST be written with performance considerations. Avoid N+1 queries, inefficient loops, and unoptimized database operations.

## 6. Agent Interaction

*   Agents will operate in designated 
lanes` (e.g., `frontend`, `backend-accounting`, `backend-user-management`).
*   Communication between agents will primarily occur through `TASKS.md` updates and code reviews.
*   If an agent encounters a blocking issue or requires clarification, it MUST update `STATUS.md` and await human intervention or guidance from another agent.
